# aware-file-system

`aware-file-system` is the change-detection layer behind `aware-cli` and the upcoming
`aware-sdk`. It watches repository folders, persists lightweight indexes, and emits
structured events so agents and automation can safely treat the filesystem as a
source of truth.

## Highlights

- **Recursive watcher** – polls directories, classifies create/update/delete events,
  and normalises them into JSON-ready payloads.
- **Index snapshots** – builds MsgPack-backed caches to accelerate large tree
  scans and cold starts.
- **Canonical source scanning** – `CanonicalSourceFilterConfig` keeps user-owned
  paths such as `demo/`, `examples/`, `docs/README.md`, `migrations/`, assets,
  and tests visible for workspace status and delta rails.
- **Pluggable handlers** – send events to aware-cli’s summary state, your own async
  consumers, or a queue service by registering callbacks.
- **Pure Python** – easy to embed inside CI, long-running daemons, or custom tools.

## Workspace SDK Contract

`aware-file-system` is the canonical local observation layer for Workspace
checkout status and delta collection. In the Workspace money lane:

```text
Local Checkout -> Deltas -> Workspace SDK -> Workspace Service
-> materialization -> apply deltas and store artifact receipts
```

this package owns filesystem truth only. It should report stable relative paths,
stat metadata, optional SHA-256 hashes, and changed-file sets. It must not infer
semantic package ownership; Workspace semantic source and explicit SDK package
specs own that mapping.

Quality requirements:

- Use `CanonicalSourceFilterConfig` for workspace status, SDK local deltas,
  commits, materialization, and local/remote sync.
- Preserve user-owned source paths, including `demo/`, `examples/`, `docs/`,
  `migrations/`, assets, tests, and README files.
- Exclude infrastructure/runtime noise such as `.git`, `.aware`, `_aware`,
  caches, virtualenvs, `node_modules`, build outputs, and generated bytecode.
- Keep path handling workspace-relative and reject traversal outside the scan
  root.
- Prefer metadata and lazy hashes for status; read file content only when a
  caller explicitly needs payloads.

## Canonical API Boundary

The canonical FileSystem contract starts at the `file-system-api` DTO package
and the `apis/filesystem` service API/protocol shell:

- `modules/file_system/structure/api/aware/file_system/types.aware`
- `modules/file_system/structure/api/aware/file_system/service_operation.aware`
- `apis/filesystem/bindings/filesystem.apis.aware`

This package should implement or adapt to those DTOs rather than defining a
separate Workspace-only filesystem contract. The intended layering is:

```text
Workspace SDK / other SDKs -> FileSystem API DTOs
-> aware-file-system Python facade/router -> Python or Rust backend execution
```

There is intentionally no FileSystem domain ontology/state model yet. The
`modules/file_system` module is API-only for this pass; local path truth,
deltas, digests, apply receipts, and backend capabilities are the public API
contract for this lane.

Longer term, `services/filesystem` can implement the generated service protocol
so Workspace and other consumers interact at an API boundary instead of importing
filesystem materialization internals. Rust should remain an execution backend
behind the same DTO receipts, not a direct dependency of Workspace SDK.

## Performance Plan

Current performance primitives are already useful: directory-level MsgPack
caches, a persistent file index, session caching, mtime/size change detection,
nanosecond mtimes, filter-result caching, and lazy SHA-256 computation. The next
work should make those guarantees explicit and measurable before rewriting.

Priority benchmark surfaces:

- `FileSystemIndex.scan_relative_metadata(...)`: total scan time, files
  processed, files content-read, cache hit ratio, changed-directory count.
- Workspace `local_fs_view`: scoped status and warm-service status should stay
  near the 1s budget; cold repo-root status must stay bounded before live agent
  sync depends on it.
- `WorkspaceLocalDeltaCollector`: changed-file to `CodePackageDelta` latency,
  especially after a small edit in a large checkout.
- Materialized output apply: batched write throughput, digest verification cost,
  and path-safety overhead.
- Artifact receipts: stable storage/index cost once artifact persistence becomes
  explicit.

Near-term Python cleanup should come first: remove duplicate full-tree scanner
paths, route the polling watcher through the same canonical index where possible,
make cache invalidation evidence visible in benchmark receipts, and add fixtures
for large unchanged trees, large scoped trees, and many small generated outputs.

The Rust apply engine now exposes a stream-capable content contract in receipts
without changing the default Python API. Compatibility service calls still use
the buffered `length_prefixed_buffered_v1` protocol, while opt-in service calls
can use `chunked_stream_v1` with `--rust-service-streaming-payload` or
`direct_chunked_stream_v1` with `--rust-service-direct-streaming-payload`. Rust reports
`content_engine.engine_kind`, `streaming_capable`, buffered/streamed payload
counts, spooled/direct payload counts, byte counts, chunk count, and max chunk
size. Streaming receipts should show `streamed_payload_count > 0` and
`bytes_buffered == 0` for the Rust apply engine; direct streaming receipts should
also show `bytes_spooled == 0` and `direct_streamed_payload_count > 0`, proving
bounded service ingestion while preserving path safety, before/after
observations, digest verification, and artifact counters inside Rust apply.

## Native Backend Direction

Rust is a good candidate for the first native acceleration layer, but only behind
the existing Python API. The package contract should remain `aware-file-system`
with a pure Python fallback and an optional native backend for hot paths.
Longer term, this package should become the Python service-protocol
router/facade for a canonical FileSystem API/Service contract, while Rust owns
the native filesystem engine implementation for observation, hashing, cache, and
apply operations. Workspace SDK should keep talking to Workspace/API surfaces,
not directly to Rust.

The native work is split deliberately:

- Python exposes `aware_file_system.native_backend` for fallback-first capability
  detection. Importing `aware_file_system` must not require a native module.
- Rust lives under `workspaces/aware_kernel/modules/filesystem/libs/file_system/rust/aware_file_system_native`. It reports
  `capability_report`, parity-only `workspace_snapshot`, and parity-only
  `workspace_hashes`, and parity-only `workspace_apply_deltas` operations. The
  crate is feature-owned: capabilities, filtering, observation, hashing, apply,
  shared digest, and shared JSON encoding are separate modules so
  cache/service bindings can grow without a monolithic native surface.
- Python exposes `aware_file_system.native_snapshot` as an explicit dev/parity
  bridge that can invoke the Rust CLI and compare it against the Python
  `CanonicalSourceFilterConfig` scanner.
- Python exposes `aware_file_system.native_hash` as the matching SHA-256 parity
  bridge. It compares Rust `workspace_hashes` output against Python
  `hashlib.sha256` for the same selected changed paths and checks the v0
  cache-counter receipt shape.
- Python exposes `aware_file_system.native_apply` as the materialized file apply
  parity bridge. It compares Rust path-safe create/update/delete receipts
  against a Python implementation, including before/after SHA-256 values,
  byte counters, and digest-verification counts. Artifact blob storage is not a
  v0 side effect; apply receipts explicitly report zero stored artifacts.
- Python exposes `aware_file_system.native_apply_executor` as the single Rust
  apply execution surface. It prepares the CLI through `rust-tooling` or accepts
  an already prepared binary, then both benchmark and selector paths use the same
  executor shape. It also owns the persistent apply-service proof used only by
  performance receipts before any native-library or Workspace SDK routing move.
- Native availability is not a performance claim. The native backend must first
  emit `aware.file_system.workspace_fs_benchmark.v1` receipts and pass
  `assert_workspace_fs_benchmark_parity(...)` against Python receipts.
- The Workspace SDK and normal Python scanner do not route through Rust until
  parity has covered walk/filter/snapshot, hash/cache, and apply behavior.

Best first Rust candidates:

- directory walking with canonical ignore/filter evaluation;
- metadata snapshot diffing and cache serialization;
- streaming SHA-256 for many files;
- batched materialized-output writes with digest verification;
- future cross-platform watcher adapters.

Do not migrate semantic package matching, Workspace lifecycle decisions, or SDK
API orchestration into Rust. Those remain Workspace/API responsibilities. Native
code should make filesystem observation and apply operations faster without
changing which paths are considered canonical.

## Filter Profiles

Use `CanonicalSourceFilterConfig` for authoritative source observation: workspace
status, SDK local deltas, commits, and semantic materialization. It excludes
infrastructure/runtime noise only.

Use `CodeIntrospectionFilterConfig` only for optional noise-reduced
introspection/summarization. It is heuristic and must not be used to decide
whether a workspace is clean or to produce canonical source deltas.

```bash
pip install aware-file-system
```

```bash
pip install "aware-file-system[native]"
```

Requires Python 3.12 or newer.

## Quick start

```python
import asyncio
from aware_file_system.config import Config, FileSystemConfig
from aware_file_system.models import ChangeType
from aware_file_system.watcher.file_system_watcher import FileSystemWatcher

async def main() -> None:
    config = Config(file_system=FileSystemConfig(root_path="docs/projects"))
    watcher = FileSystemWatcher(config, poll_interval=2.0)

    def handle(event) -> None:
        if event.change_type is ChangeType.MODIFIED:
            print(f"{event.path} updated ({event.checksum.sha256[:8]})")

    watcher.add_event_handler(handle)

    await watcher.start()
    try:
        await asyncio.sleep(10)
    finally:
        await watcher.stop()

asyncio.run(main())
```

### Integrating with aware-cli

```bash
aware-cli summary \
  --project aware-sdk \
  --task cli-release-bundles \
  --watch-root docs/projects
```

The CLI wires `aware-file-system` under the hood to stream change events into its
summary state. You can reuse the same configuration helpers (environment variables,
YAML config) to point at other workspaces or embed the watcher inside your tooling.

## Configuration reference

- `AWARE_FILE_SYSTEM_ROOT` – override the root directory that should be scanned.
- `AWARE_FILE_SYSTEM_POLL_INTERVAL` – polling cadence in seconds (default: 2.0).
- `AWARE_FILE_SYSTEM_CACHE_DIR` – location for MsgPack indexes (default: `.aware/fs`).
- `aware_file_system.config.Config` – Python API with sensible defaults and
  room for custom callbacks.
